嵌入式软件设计方法学习笔记(Edit by HNU BJB)

第一章：面向对象编程

1.1 基础概念

  • 对象：数据与函数的组合，由一组描述对象状态的数据和一组对象支持操作的函数构成。在 C++ 中，对象通常具有封装、继承和多态三个核心特性。

  • 类：最常见的对象实现方式，包含属性和方法。属性可以理解为结构体成员、参数等；方法可以理解为针对对象进行操作的函数。

  • UML 类图：用于描述类与类之间关系的结构图。在 Markdown 中，可以使用 Mermaid 实现简单的类图。

1.2 三大特性

1.2.1 封装

  • 名词解释：将属性和方法封装起来，属性只能通过对外提供的方法访问，内部实现细节对外不可见。

  • 编程案例：在 C 语言中，常见封装方式是将头文件和实现文件分开：头文件只暴露结构体名、函数名和必要参数，不暴露内部成员。

1.2.2 继承

  • 名词解释：定义一个子类，继承父类的属性和方法，并在此基础上扩展。

  • 编程案例：在 C 语言中，可以通过在子结构体中包含父结构体对象的方式模拟继承。

1.2.3 多态

  • 名词解释：针对不同对象，相同函数具有不同实现。在 C 语言中，常通过函数指针和 void * 泛型指针实现。

1.3.1 内存分配 / 对象创建

1.3.2 最少知识原则

  • 类的内部数据不管多复杂，都不能直接传递给使用者，只能通过公有方法间接访问。

  • 因此，在类的封装、继承和多态实现中，可以提供公有方法读取内部数据，而不是直接暴露内部结构。

第二章：抽象接口

2.1 抽象接口的概念

  • 抽象接口 interface：一种抽象功能接口。使用者只负责调用接口，无需知道接口内部的数据定义和功能实现方式；在 C 语言中，可以通过函数指针实现抽象接口。

2.4 依赖注入

  • 依赖注入：向某个对象提供它所需要的其他对象。常见方式包括构造注入和方法注入。

    • 构造注入：通过初始化函数完成依赖对象注入，初始化通常只调用一次。

    • 方法注入：通过某个非初始化函数注入依赖对象，可以多次调用，用于动态切换依赖对象。

  • 两种注入方式的区别:

    • 相同点：都是通过函数将依赖对象传入。
    • 不同点：
      • 联系极其紧密、初始化时必须操作依赖对象、依赖对象长期存在且不变化时，适合构造注入。
      • 依赖对象需要动态切换时，适合方法注入。

第四章：软件设计原则

4.1 SOLID 五大原则

4.1.1 S：单一职责原则

  • 名词解释：一个类只能有一个引起它变化的原因。对于函数来说，就是一个函数或类只负责一个功能，不处理多项任务。

  • 案例：录波二进制保存功能开发。原代码共计约 600 行，将多个任务放在一起，造成功能糅合、界限不清、难以拆分。如果遵循单一职责原则，应将结构中的各项功能封装为独立函数，尤其是可复用功能，例如：

    • 打开并创建文件；
    • 检查路径；
    • 检查文件句柄；
    • 数据异常检查；
    • 数据读取；
    • 数据写入。

  • 如果在此基础上增加二进制保存功能，最彻底的办法是重构代码并拆分职责；最保守的办法是使用抽象接口处理写入逻辑。

注意事项：

  • 单一职责原则不一定完全遵守，因为职责边界并不总是容易划分。
  • 对老代码的修改可能与其他原则冲突，例如开闭原则要求不修改老代码。
  • 实际工程中需要在可维护性、改动风险和开发成本之间取舍。

4.1.2 O：开闭原则

  • 名词解释：软件实体应对扩展开放，对修改关闭。也就是尽量不修改老代码，而是通过扩展新函数实现新功能。

  • 开闭原则不一定完全遵守。
  • 对老代码不轻易改动，但当重复度极高时，可以考虑重构。
  • 对新代码，应尽量预留扩展能力。

4.1.3 L：里氏替换原则

  • 名词解释：
    • 派生类必须能够替换基类。即父类对象替换为子类对象后，程序仍能正常运行。
    • 子类应包含父类相同的属性和方法，并保持行为兼容。

4.1.4 I：接口隔离原则

  • 名词解释：

    • 不应该强制客户端依赖它们不使用的接口。

    • 对于函数来说，如果函数只使用结构体极少数字段，却被迫传入完整结构体，就是接口隔离原则的反面案例。

    • 通常可以通过抽象接口分离不同类、函数或对象之间的依赖。

4.1.5 D：依赖反转原则

  • 名词解释：高层模块不应该依赖低层模块，二者都应该依赖抽象。

  • 编程案例：以录波保存为例，原函数依赖 ASCII 写入函数，不利于插入 BIN 写入函数。因此可以定义写入函数指针，根据初始化条件确定函数指针指向。

但需要注意的是：

  • 相比单一职责原则，不用大规模重构原代码；
  • 相比完全遵守开闭原则，不会出现大量重复代码；
  • 虽然同时违反部分开闭原则和单一职责原则，但改动成本更低。

原则不是天条。实际工程中更重要的是在冲突之间选择合适方案。

4.2 七大耦合：由强到弱

4.2.1 内容耦合

名词解释：模块直接访问另一个模块的内部数据或指令。

4.2.2 公共耦合

名词解释：多个模块访问同一个全局数据区。

4.2.3 外部耦合

名词解释：模块访问同一个外部定义的数据格式、通信协议等。

4.2.4 控制耦合

名词解释：一个模块通过传递控制信息来控制另一个模块的行为。

4.2.5 标记耦合

名词解释：模块间传递数据结构，但只使用其中部分字段。

4.2.6 数据耦合

名词解释：模块间只传递简单数据。

4.2.7 无直接耦合

名词解释：模块之间完全独立，通过参数列表或外部调度者交换信息。
